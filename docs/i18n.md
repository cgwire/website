# Gestion des locales (EN / FR / JA)

## Où ça vit

| Fichier | Rôle |
|---|---|
| `nuxt.config.ts` (bloc `i18n`) | Config du module `@nuxtjs/i18n` : 3 locales, `defaultLocale: en` (sans préfixe), `lazy: true`, `customRoutes: 'config'`, `detectBrowserLanguage: false` |
| `i18n/routes.js` | Les chemins d'URL traduits par route nommée (`/about` → `/a-propos` → `/企業情報`) |
| `i18n/locales/{en,fr,ja}.json` | Les chaînes d'UI (header, footer, pricing, faq-section...) plus le bloc `slugs.*` |
| `content/{en,fr,ja}_pages.json` | Le contenu structuré, une copie complète par langue (pages, features, alternatives, audiences, studios, testimonials, customer_stories) |
| `content/{pages,faq,tools}/{en,fr,ja}/*.md` | Le contenu markdown, un dossier par langue |
| `app/composables/useI18NSlug.js` | La colle : résout le slug d'URL localisé vers le slug canonique EN |

## Les trois couches de traduction

1. **Chaînes d'UI** : `$t('header.pricing')` résolu dans `i18n/locales/{lang}.json`, chargé paresseusement. Même arborescence de clés dans les 3 fichiers.
2. **Chemins d'URL** : `i18n/routes.js` est passé à `i18n.pages`. Le même `app/pages/pricing.vue` sert `/pricing`, `/fr/tarifs`, `/ja/料金体系`. Les routes dynamiques sont des gabarits (`'features-slug': { fr: '/fonctionnalités/[slug]' }`).
3. **Contenu** : jamais dans les fichiers de locale. Requêté via Nuxt Content avec `.where('lang', '=', locale.value)`.

## Le point subtil : `slugs.*` est bidirectionnel

Dans les JSON de contenu, le champ `slug` reste toujours le slug canonique anglais, y compris dans `fr_pages.json` et `ja_pages.json`. Les fichiers de locale portent donc les deux sens :

- `slugs.casting-management` → `gestion-des-castings` (EN vers FR, pour construire l'URL)
- `slugs.gestion-des-castings` → `casting-management` (FR vers EN, pour retrouver le contenu)

À l'arrivée sur `/fr/fonctionnalités/gestion-des-castings`, `useI18NSlug` fait le chemin retour avant la requête :

```js
queryCollection('jsonPages')
  .where('lang', '=', 'fr')                 // quelle traduction
  .where('slug', '=', 'casting-management') // toujours la clé EN
```

Dans l'autre sens, il pousse `setI18nParams({ en, fr, ja })` pour que `$switchLocalePath()` du `LanguageSwitcher` produise le bon slug dans chaque langue. `te()` fait un fallback silencieux : les slugs de studios n'ont pas d'entrée et restent identiques partout.

## Au build

`buildPrerenderRoutes()` dans `nuxt.config.ts` reconstruit la liste complète des routes à prérendre, parce que le crawler rate les pages non liées. Il lit `content/en_pages.json` et les noms de fichiers md, applique `locales[lang].slugs[slug]`, applique le gabarit de `i18n/routes.js`, puis préfixe `/fr` ou `/ja`. `failOnError: true` fait échouer le build plutôt que de laisser disparaître une page.

## En pratique

- Nouveau slug traduit : 2 entrées à ajouter dans `fr.json` et `ja.json` (aller et retour), sinon l'URL FR renvoie un 404 côté contenu.
- Nouvelle route : entrée dans `i18n/routes.js`, plus une entrée dans `buildPrerenderRoutes` si elle n'est liée depuis nulle part.
- Édition d'un `*_pages.json` : pas de hot-reload, faire `rm -rf .nuxt .data && npm run dev`.
